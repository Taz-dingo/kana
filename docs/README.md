# Kana Docs

这套文档是 `kana` 项目的长期工作台，用来沉淀项目的目标、设计哲学、当前状态、参考案例和下一步任务。

`kana` 已经是独立仓库，所以这里的文档只服务这个项目本身，不再考虑 monorepo 的边界问题。

## 组织方式

- `spec.md`
  - 回答“这个项目为什么存在、希望做成什么样、哪些事情不做”
  - 适合记录产品定位、学习体验目标、设计哲学、视觉原则、非目标
- `plan.md`
  - 回答“接下来一段时间怎么推进”
  - 适合记录阶段拆分、优先级、依赖关系、待确认决策
- `status.md`
  - 回答“项目现在到什么程度了”
  - 适合记录已完成、进行中、已知问题、最近变化
- `tasks.md`
  - 回答“现在具体要做什么”
  - 适合记录可执行任务、优先级、状态、下一步落地项
- `references.md`
  - 回答“外部参考里哪些点值得借鉴、哪些不能照搬”
  - 适合记录竞品观察、信息架构拆解、交互启发
- `memory-research.md`
  - 回答“间隔重复、Anki、FSRS 与遗忘研究对这个项目意味着什么”
  - 适合沉淀记忆系统研究结论、科学边界与设计启发
- `memory-system.md`
  - 回答“如果把 `kana` 做成带记忆调度的学习产品，系统应该怎么设计”
  - 适合记录数据模型、调度逻辑、页面结构与落地顺序

## 维护约定

- `spec.md` 低频更新，只在产品方向、设计原则、学习模型变化时调整
- `plan.md` 中频更新，只要阶段目标或推进顺序变化就更新
- `status.md` 高频更新，每完成一轮明显工作后刷新
- `tasks.md` 持续更新，保持任务短、具体、可执行
- `references.md` 按需更新，避免堆砌灵感链接，只记录真正影响设计决策的观察
- `memory-research.md` 在研究结论发生变化或补充新证据时更新
- `memory-system.md` 在记忆系统设计方向、数据模型或阶段目标变化时更新

## 当前原则

- 文档优先为设计和开发决策服务，不写空泛口号
- 先讲清楚学习目标，再决定 UI 形式和技术选型
- 保持短文档、强约束、可迭代，避免一次性写得太满
- 后续获取上下文时，默认直接阅读 `spec.md`、`status.md`、`tasks.md`，需要研究背景时补读 `memory-research.md` 与 `memory-system.md`
